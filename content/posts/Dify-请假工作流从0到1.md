# Dify 实战：从零搭建智能请假申请工作流（RAG + 条件分支 + 多轮对话）

> 本文记录了从零开始在 Dify 平台上搭建一个完整的"智能请假申请助手"的全过程，涵盖知识库构建、Chatflow 工作流设计、条件分支、多轮对话补全信息等核心功能。
>
> 适合 Dify 新手入门参考。

---

## 一、项目背景

最近领导安排了一个任务：跑通 Dify 平台的三个核心功能——**工作流**、**Agent**、**API**。

Dify 是一个开源的 LLM 应用开发平台，提供了可视化的工作流编排、RAG 管道、Agent 能力等功能，可以快速构建基于大模型的应用。

之前尝试过一次请假申请单工作流，但效果不太理想，于是决定从头来过，系统性地走一遍完整流程。

## 二、技术环境

| 组件 | 配置 |
|------|------|
| Dify 版本 | 1.14.2 |
| 部署方式 | Docker Compose（12 个容器） |
| 访问地址 | http://localhost:8080 |
| 数据库 | PostgreSQL 15 |
| 向量库 | Weaviate |
| LLM 模型 | Ollama（本地）+ 百炼 DeepSeek（阿里云） |
| Embedding 模型 | bge-m3（本地 Ollama） |

## 三、整体方案设计

### 3.1 场景描述

用户通过自然语言描述请假需求，系统自动：
1. 从考勤制度知识库中检索相关规定
2. 提取请假关键信息（类型、时间、原因）
3. 判断信息是否完整
4. 完整 → 生成标准请假单
5. 不完整 → 提示用户补充缺失信息

### 3.2 工作流架构

```
┌──────────────┐
│    START     │  用户输入请假请求
│  (用户输入)   │  例："我想请三天假"
└──────┬───────┘
       │
       ▼
┌──────────────────────┐
│ KNOWLEDGE_RETRIEVAL  │  从考勤制度知识库检索相关规定
│ (考勤制度知识库)       │
└──────┬───────────────┘
       │
       ▼
┌──────────────────────┐
│        LLM           │  分析用户请求 + 检索结果
│  (信息提取与分析)      │  提取：请假类型、时间、原因等
└──────┬───────────────┘
       │
       ▼
┌──────────────────────┐
│      IF_ELSE         │  条件判断：信息是否完整？
└──┬───────────────┬───┘
   │               │
   ▼               ▼
[完整]          [缺少信息]
   │               │
   ▼               ▼
┌──────────┐  ┌──────────────┐
│   LLM    │  │     LLM      │
│(生成请假单)│  │(提示缺少的字段)│
└────┬─────┘  └──────┬───────┘
     │               │
     ▼               ▼
┌──────────┐  ┌──────────────┐
│  ANSWER  │  │    ANSWER    │
│(输出请假单)│  │(请用户补充信息)│
└──────────┘  └──────┬───────┘
                     │
                     ▼
              用户回复补充信息
              └──→ 再次进入流程
```

### 3.3 应用类型选择

Dify 支持多种应用类型：

| 类型 | 特点 | 适用场景 |
|------|------|---------|
| Chat | 简单对话 | 单轮问答 |
| **Chatflow** | **高级对话编排** | **多轮对话、需要条件分支** |
| Workflow | 工作流 | 单次执行、无需对话 |
| Agent | 智能体 | 自主决策、工具调用 |

本场景需要**多轮对话**（用户补充信息后继续处理），因此选择 **Chatflow**。

## 四、阶段 1：知识库构建

### 4.1 准备考勤制度文档

首先准备一份考勤管理制度文档，内容涵盖请假类型、流程、处罚等：

```
公司考勤管理制度

第一章 总则
...

第三章 请假类型
第五条 事假
1. 因个人事务需要请假的，可申请事假。
2. 事假每月累计不超过 3 天，每年累计不超过 15 天。
3. 事假期间不发放工资。

第六条 病假
1. 因病需要休息治疗的，可申请病假。
2. 病假需提供二级及以上医院出具的诊断证明和病假条。
...

第四章 请假流程
第十二条 请假审批权限
1. 请假 1 天以内（含 1 天）：由直属主管审批。
2. 请假 1 天以上 3 天以内（含 3 天）：由部门经理审批。
3. 请假 3 天以上：由总经理审批。
...
```

### 4.2 创建知识库

1. Dify 左侧菜单 → **知识库** → **创建知识库**
2. 名称：`考勤管理制度`
3. 上传文档（支持 Word/TXT/PDF）

### 4.3 配置 Embedding 模型

本地 Ollama 已部署 `bge-m3` 模型，在 Dify 中配置：

- **设置** → **模型供应商** → **Ollama** → **添加模型**
- 模型名称：`bge-m3`
- 基础 URL：`http://host.docker.internal:11434`
- 模型类型：Text Embedding

> **注意**：Docker 容器内访问宿主机 Ollama，使用 `host.docker.internal` 地址。

### 4.4 分段与索引

- 分段标识符：`\n\n`（按段落分段）
- 最大分段长度：500
- 索引方式：**高质量**（使用 Embedding 模型向量化）
- 检索模式：**混合检索**（语义检索 + 全文检索，权重 0.7:0.3）
- TopK：3

### 4.5 召回测试

上传完成后，进行召回测试验证：

| 测试查询 | 预期结果 | 状态 |
|---------|---------|------|
| "请假需要什么材料？" | 检索到病假需提供医院证明 | ✅ |
| "年假有几天？" | 检索到年假 5/10/15 天 | ✅ |
| "迟到怎么处罚？" | 检索到迟到扣款规则 | ✅ |

## 五、阶段 2：工作流搭建

### 5.1 创建 Chatflow 应用

1. **工作室** → **创建应用** → **Chatflow**
2. 名称：`请假申请助手`

### 5.2 配置开始节点

开始节点默认有两个变量：
- `userinput.query` — 用户输入的文本
- `userinput.files` — 用户上传的文件

保持默认即可。

### 5.3 添加知识检索节点

从开始节点向下拖拽，添加 **知识检索** 节点：

| 配置项 | 值 |
|--------|-----|
| 知识库 | 考勤管理制度 |
| 查询变量 | sys.query（用户输入） |
| 检索模式 | 混合检索 |
| TopK | 3 |

### 5.4 配置 LLM 节点（信息提取）

从知识检索节点向下拖拽，添加 **LLM** 节点：

**模型选择**：百炼 DeepSeek（复杂提取任务用强模型）

**SYSTEM Prompt**：

```
你是一个请假申请助手。你的任务是根据用户的请求和考勤制度知识库的内容，提取请假所需的关键信息。

请提取以下字段：
1. 请假类型（事假/病假/年假/调休/婚假/产假/丧假）
2. 请假开始时间
3. 请假结束时间
4. 请假天数
5. 请假原因

如果某项信息用户未提供，请标记为"缺失"。

考勤制度参考信息：
{{#context#}}

用户请求：{{#query#}}

请严格以以下 JSON 格式输出，不要输出其他内容：
{
  "leave_type": "请假类型或缺失",
  "start_date": "开始时间或缺失",
  "end_date": "结束时间或缺失",
  "days": "天数或缺失",
  "reason": "原因或缺失",
  "is_complete": true或false,
  "missing_fields": ["缺失的字段名列表"]
}
```

**关键配置**：LLM 节点的 **"上下文"** 选项栏，选择 **知识检索节点的 result**，这样检索到的内容会注入到 `{{#context#}}` 占位符中。

### 5.5 添加条件分支节点（IF_ELSE）

从 LLM 节点向下拖拽，添加 **条件分支（IF/ELSE）** 节点：

| 配置项 | 值 |
|--------|-----|
| 变量 | LLM 节点 → text |
| 条件类型 | 包含 |
| 值 | `"is_complete": true` |

- **IF 分支**：信息完整 → 生成请假单
- **ELSE 分支**：信息缺失 → 提示补充

### 5.6 IF 分支：生成请假单

从 IF_ELSE 的 **IF 端口** 向下拖拽，添加 **LLM** 节点：

**SYSTEM Prompt**：

```
你是一个请假申请助手。当前日期是2026年6月。请根据以下信息生成一份格式规范的请假申请单。

用户原始请求：{{#query#}}

提取的请假信息：{{#信息提取LLM节点.text#}}

考勤制度参考：{{#context#}}

请生成一份正式的请假申请单，包含以下内容：
- 标题：请假申请单
- 申请人信息（如未提供姓名，用"待填写"代替）
- 请假类型
- 请假时间（起止日期）
- 请假天数
- 请假原因
- 审批流程说明（根据考勤制度中的规定）
- 申请日期

格式要求：使用表格或对齐的文本格式，清晰易读。
```

> **注意**：`{{#信息提取LLM节点.text#}}` 需要通过变量选择器选取实际的 LLM 节点变量，不要手动输入。

然后添加 **回答** 节点，输出选择该 LLM 的 text。

### 5.7 ELSE 分支：提示补充信息

从 IF_ELSE 的 **ELSE 端口** 向下拖拽，添加 **LLM** 节点：

**SYSTEM Prompt**：

```
你是一个友好的请假申请助手。用户提交的请假申请缺少一些必要信息，请礼貌地告知用户需要补充哪些内容。

用户原始请求：{{#query#}}

提取的请假信息：{{#信息提取LLM节点.text#}}

考勤制度参考：{{#context#}}

请根据以上信息：
1. 告知用户已经获取到的信息（如有）
2. 明确列出需要补充的字段，并给出具体的提示或示例
3. 语气友好，引导用户补充完整信息
```

然后添加 **回答** 节点，输出选择该 LLM 的 text。

### 5.8 最终工作流结构

```
开始 (userinput.query)
  │
  ▼
知识检索 (考勤管理制度)
  │
  ▼
LLM (信息提取)
  │
  ▼
条件分支 (IF/ELSE)
  │           │
  ▼           ▼
[IF/完整]   [ELSE/缺失]
  │           │
  ▼           ▼
LLM         LLM
(生成请假单)  (提示补充)
  │           │
  ▼           ▼
回答         回答
(请假单)     (补充提示)
```

## 七、踩坑与经验

### 7.1 系统变量中没有 current_date

Dify 的系统变量不包含当前日期。可用的系统变量有：
- `sys.query` — 用户输入
- `sys.files` — 上传的文件
- `sys.user_id` — 用户 ID
- `sys.conversation_id` — 会话 ID
- `sys.timestamp` — Unix 时间戳（数字格式）

**解决方案**：在 Prompt 中硬编码当前日期，如 `当前日期是2026年6月`。

### 7.2 Docker 容器内访问宿主机服务

Dify 运行在 Docker 中，访问宿主机的 Ollama 服务需要使用特殊地址：
- Windows/Mac：`http://host.docker.internal:11434`
- Linux：需要在 docker-compose 中添加 `extra_hosts` 配置

### 7.3 上下文变量的配置

LLM 节点有一个专门的 **"上下文"** 选项栏，用于注入知识库检索结果。选择知识检索节点的 `result` 变量后，Prompt 中的 `{{#context#}}` 占位符会被自动替换。

### 7.4 变量引用格式

在 Dify 工作流中引用其他节点的变量时，使用 `{{#节点名.变量名#}}` 格式。建议通过变量选择器选取，避免手动输入出错。

## 九、总结

通过这个实战项目，我们走通了 Dify 平台的核心功能：

1. **知识库（RAG）**：上传文档 → 分段 → 向量化 → 混合检索
2. **工作流编排**：START → 知识检索 → LLM → 条件分支 → 回答
3. **条件分支**：根据 LLM 输出判断走哪个分支
4. **多轮对话**：Chatflow 模式支持用户补充信息后继续处理

Dify 的可视化工作流编排降低了 LLM 应用开发的门槛，让开发者可以专注于业务逻辑和 Prompt 设计，而不需要写大量的胶水代码。
