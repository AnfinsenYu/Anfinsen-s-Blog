# Dify 实战：从零搭建智能请假申请工作流

> 本文是 Dify 实战系列的第二篇，详细介绍如何使用 Dify 的 Chatflow 功能搭建一个智能请假申请工作流。
>
> 该工作流支持：知识库检索、信息提取、条件判断、多轮对话补全信息等功能。

---

## 一、项目背景

领导交代的任务是验证 Dify 平台的三个核心功能：**工作流、Agent、API**。

本文聚焦第一个功能——**工作流**，通过搭建一个「智能请假申请系统」来验证 Dify 的 Chatflow 能力。

### 需求概述

用户通过自然语言描述请假需求，系统自动：
1. 检索公司考勤制度
2. 提取请假关键信息（类型、时间、原因）
3. 判断信息完整性
4. 生成标准请假单 或 提示用户补充信息

---

## 二、工作流设计

### 2.1 应用类型选择

Dify 提供多种应用类型：

| 类型 | 说明 | 适用场景 |
|------|------|----------|
| **Chat** | 简单对话 | 单轮问答 |
| **Chatflow** | 对话式工作流 | 需要多轮对话、条件分支 |
| **Workflow** | 工作流 | 后台批处理、无对话 |
| **Agent** | 智能代理 | 动态决策、工具调用 |
| **Completion** | 文本补全 | 文本生成 |

**选择 Chatflow**，因为：
- 需要多轮对话（用户补充信息）
- 需要条件分支（信息完整/不完整）
- 需要知识库检索（考勤制度）

### 2.2 流程图

```
┌──────────────┐
│    START     │  用户输入请假请求
│  (用户输入)   │  例："我想请三天假"
└──────┬───────┘
       │
       ▼
┌──────────────────────┐
│ KNOWLEDGE_RETRIEVAL  │  从考勤制度知识库检索相关规定
│ (考勤制度知识库)       │
└──────┬───────────────┘
       │
       ▼
┌──────────────────────┐
│        LLM           │  分析用户请求 + 检索结果
│  (信息提取与分析)      │  提取：请假类型、时间、原因等
└──────┬───────────────┘  输出结构化信息 + 完整性判断
       │
       ▼
┌──────────────────────┐
│      IF_ELSE         │  条件判断
│  (信息是否完整？)      │
└──┬───────────────┬───┘
   │               │
   ▼               ▼
[完整]          [缺少信息]
   │               │
   ▼               ▼
┌──────────┐  ┌──────────────┐
│   LLM    │  │     LLM      │
│(生成请假单)│  │(提示缺少的字段)│
└────┬─────┘  └──────┬───────┘
     │               │
     ▼               ▼
┌──────────┐  ┌──────────────┐
│  ANSWER  │  │    ANSWER    │
│(输出请假单)│  │(请用户补充信息)│
└──────────┘  └──────┬───────┘
                     │
                     ▼
              用户回复补充信息
              └──→ 再次进入流程
```

## 三、知识库准备

### 3.1 创建考勤制度文档

首先准备一份考勤管理制度文档，内容包括：

```markdown
公司考勤管理制度

一、工作时间
1. 实行标准工时制，每周工作五天，休息两天
2. 上午 9:00-12:00，下午 13:30-18:00
3. 员工应按时上下班，不得无故迟到、早退

二、请假类型
1. 事假：因个人事务需要请假，每月不超过3天，每年不超过15天
   - 事假期间不发放工资
   - 需提前1个工作日申请
2. 病假：因病需要休息
   - 需提供二级及以上医院诊断证明和病假条
   - 可事后补办，需在3个工作日内提交材料
   - 病假工资按当地最低工资标准的80%发放
3. 年假：带薪年休假
   - 满1年不满10年：5天
   - 满10年不满20年：10天
   - 满20年：15天
   - 需提前1个工作日申请
4. 婚假：法定婚假3天
   - 需提供结婚证复印件
   - 领取结婚证后6个月内使用
5. 产假：
   - 女员工：98天（难产增加15天，多胞胎每多一个增加15天）
   - 男员工陪产假：15天
6. 丧假：
   - 直系亲属（父母、配偶、子女）：3天
   - 祖父母、外祖父母：1天
7. 调休：因加班产生的调休
   - 需提前1个工作日申请
   - 加班后3个月内使用完毕

三、请假流程
1. 员工填写请假申请单，注明请假类型、时间、原因
2. 提交至相应审批人审批
3. 审批通过后，将请假单交人力资源部备案

四、考勤处罚
1. 迟到、早退30分钟以内：每次扣款50元
2. 迟到、早退超过30分钟：按事假半天处理
3. 当月累计迟到、早退3次以上：给予书面警告
4. 旷工1天扣发3天工资
5. 连续旷工3天或年度累计旷工5天以上：公司有权解除劳动合同
```

### 3.2 创建知识库

1. 在 Dify Web UI 点击左侧「知识库」
2. 点击「创建知识库」
3. 填写信息：
   - 名称：`考勤管理制度`
   - 描述：`公司考勤管理制度文档，包含请假类型、流程、处罚等信息`
4. 上传文档（支持 Word/TXT/PDF）
5. 等待索引完成（状态变为「可用」）

### 3.3 知识库配置

| 配置项 | 推荐值 | 说明 |
|--------|--------|------|
| 索引模式 | 高质量 | 使用 Embedding 模型 |
| 检索模式 | 语义检索 | 基于语义相似度 |
| Top K | 3 | 返回最相关的 3 个片段 |
| Embedding 模型 | Ollama bge-m3 | 本地 Embedding 模型 |

> **注意**：如果使用 Ollama 作为 Embedding 模型，请确保 Ollama 服务已启动且已下载 `bge-m3` 模型。

## 四、工作流搭建

### 4.1 创建 Chatflow 应用

1. 在 Dify Web UI 点击「创建应用」
2. 选择「Chatflow」类型
3. 填写信息：
   - 名称：`智能请假申请`
   - 描述：`基于考勤知识库的智能请假申请系统`

### 4.2 配置节点

#### 节点 1: START（开始节点）

- 类型：开始节点
- 输入变量：
  - `query`（文本）：用户的请假请求

#### 节点 2: KNOWLEDGE_RETRIEVAL（知识检索）

- 类型：知识检索
- 知识库：考勤管理制度
- 检索模式：语义检索
- Top K：3
- 输出变量：`context`（检索到的相关内容）

#### 节点 3: LLM（信息提取）

- 类型：LLM
- 模型：DeepSeek（百炼）
- 输入变量：
  - `query`：用户输入
  - `context`：知识库检索结果
- Prompt：

```
你是一个请假申请助手。根据用户的请求和考勤制度，提取以下信息：
1. 请假类型（事假/病假/年假/调休/婚假/产假等）
2. 请假开始时间
3. 请假结束时间
4. 请假天数
5. 请假原因

如果某项信息用户未提供，标记为"缺失"。

考勤制度参考：
{{#context#}}

用户请求：{{#query#}}

请以 JSON 格式输出：
{
  "leave_type": "请假类型或缺失",
  "start_date": "开始时间或缺失",
  "end_date": "结束时间或缺失",
  "days": "天数或缺失",
  "reason": "原因或缺失",
  "is_complete": true/false,
  "missing_fields": ["缺失的字段列表"]
}
```

- 输出变量：`extract_result`（JSON 格式的提取结果）

#### 节点 4: IF_ELSE（条件分支）

- 类型：条件分支
- 条件：判断 `extract_result` 中的 `is_complete` 字段
  - 如果 `is_complete` == `true` → 完整分支
  - 如果 `is_complete` == `false` → 缺失分支

#### 节点 5: LLM（生成请假单）— 完整分支

- 类型：LLM
- 模型：DeepSeek（百炼）
- 输入变量：
  - `extract_result`：提取结果
  - `context`：知识库检索结果
- Prompt：

```
根据以下信息生成一份正式的请假申请单：

请假信息：{{#extract_result#}}
考勤制度：{{#context#}}

请生成格式规范的请假单，包含：
1. 标题：请假申请单
2. 申请人信息（如有）
3. 请假详情（类型、时间、天数、原因）
4. 审批流程说明（根据考勤制度）

使用清晰的格式输出。
```

- 输出变量：`leave_form`（生成的请假单）

#### 节点 6: LLM（提示补充）— 缺失分支

- 类型：LLM
- 模型：DeepSeek（百炼）
- 输入变量：
  - `extract_result`：提取结果
- Prompt：

```
用户的请假申请缺少以下信息：

{{#extract_result.missing_fields#}}

请友好地告知用户需要补充哪些信息，并给出具体的提示。
例如：
- 如果缺少请假时间，请提醒用户说明具体的开始和结束日期
- 如果缺少请假类型，请提醒用户说明是事假、病假、年假等
- 如果缺少请假原因，请提醒用户简要说明原因

语气要友好、专业。
```

- 输出变量：`prompt_message`（补充信息提示）

#### 节点 7: ANSWER（输出请假单）— 完整分支

- 类型：回答节点
- 输出变量：`leave_form`（生成的请假单）

#### 节点 8: ANSWER（提示补充）— 缺失分支

- 类型：回答节点
- 输出变量：`prompt_message`（补充信息提示）

### 4.3 连接节点

按照流程图连接各节点：

```
START → KNOWLEDGE_RETRIEVAL → LLM(信息提取) → IF_ELSE
                                                ├─ 完整 → LLM(生成请假单) → ANSWER(请假单)
                                                └─ 缺失 → LLM(提示补充) → ANSWER(提示)
```

## 六、关键知识点

### 6.1 Chatflow vs Workflow

| 特性 | Chatflow | Workflow |
|------|----------|----------|
| 对话能力 | 支持多轮对话 | 无对话，单次执行 |
| 用户交互 | 需要用户输入 | 无需用户输入 |
| 适用场景 | 客服、问答、申请 | 数据处理、批处理 |
| 会话管理 | 自动管理 conversation_id | 无会话概念 |

### 6.2 知识库检索

Dify 的知识库检索基于 RAG（Retrieval-Augmented Generation）：

1. **文档分段**：将文档拆分为多个片段
2. **向量化**：使用 Embedding 模型将片段转换为向量
3. **存储**：向量存储在 Weaviate 等向量数据库中
4. **检索**：用户查询向量化后，与知识库向量进行相似度匹配
5. **返回**：返回最相关的 Top K 个片段

### 6.3 条件分支

IF_ELSE 节点支持多种条件判断：

- **字符串比较**：等于、不等于、包含、不包含
- **数字比较**：等于、大于、小于、大于等于、小于等于
- **布尔判断**：true/false
- **JSON 字段判断**：从 JSON 中提取字段进行判断

### 6.4 变量引用

在 Dify 中，变量使用 `{{#variable_name#}}` 格式引用：

- `{{#query#}}`：用户输入
- `{{#context#}}`：知识库检索结果
- `{{#node_name.output_variable#}}`：其他节点的输出

## 七、常见问题

### Q1: 知识库检索失败

**现象**：检索结果为空或不相关

**解决**：
1. 检查 Embedding 模型是否配置正确
2. 确认知识库索引状态为「可用」
3. 尝试调整 Top K 值
4. 检查文档内容是否清晰

### Q2: LLM 输出格式不正确

**现象**：LLM 没有输出预期的 JSON 格式

**解决**：
1. 在 Prompt 中明确指定输出格式
2. 使用 Few-shot 示例
3. 尝试不同的模型

### Q3: 条件分支判断失败

**现象**：IF_ELSE 节点没有进入预期的分支

**解决**：
1. 检查条件表达式是否正确
2. 确认变量名是否正确
3. 查看节点输出，确认数据格式

### Q4: 多轮对话上下文丢失

**现象**：后续对话没有继承之前的上下文

**解决**：
1. 确保使用 Chatflow 类型（不是 Workflow）
2. 检查 conversation_id 是否正确传递
3. 确认 ANSWER 节点配置正确

## 八、总结

通过本文，我们完成了一个完整的智能请假申请工作流：

1. **知识库准备**：上传考勤制度文档，创建知识库
2. **工作流搭建**：配置 START → 知识检索 → LLM → 条件分支 → ANSWER
3. **测试验证**：测试完整请假、信息缺失、多轮对话等场景

### 核心功能

| 功能 | 实现方式 |
|------|----------|
| 知识库检索 | KNOWLEDGE_RETRIEVAL 节点 |
| 信息提取 | LLM 节点 + JSON 格式 Prompt |
| 条件判断 | IF_ELSE 节点 |
| 多轮对话 | Chatflow 自动管理 |
| 生成输出 | LLM 节点 + ANSWER 节点 |
