# 从零到一：用 RAGFlow 搭建企业考勤助手

> 基于 RAGFlow + Ollama + DeepSeek 的企业知识问答系统实战

---

## 一、项目背景

### 1.1 业务需求

公司 HR 部门每天都会收到大量考勤相关的重复咨询：
- "年假还剩几天？"
- "迟到扣多少钱？"
- "加班怎么申请调休？"

这些问题的答案都在《考勤管理制度》文档里，但员工不会主动去读。我们需要一个**智能问答助手**，让员工直接提问就能获得准确答案。

### 1.2 技术选型

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| Dify | 简单易用，部署快 | RAG 能力较弱 | ❌ |
| RAGFlow | 深度文档理解，检索精度高 | 部署复杂，速度较慢 | ✅ |
| 自研 | 完全可控 | 开发成本高 | ❌ |

**选择 RAGFlow 的原因：**
- 支持复杂文档结构（表格、多级标题）
- 内置 Rerank 机制，检索更准确
- 可视化管理界面，方便后续维护

### 1.3 最终效果

```
👤 员工：迟到 15 分钟以内怎么处理？

🤖 考勤助手：根据《考勤管理制度》第 3.2 条：
   - 迟到 15 分钟以内，每次扣款 50 元
   - 月累计 3 次及以上，额外扣除当月绩效 10%
   - 建议提前规划出行时间，避免高峰期
```

### 2.2 安装 Ollama

```bash
# 1. 下载安装 Ollama
# 访问 https://ollama.com 下载 Windows 版本

# 2. 拉取模型
ollama pull deepseek-r1:1.5b
ollama pull bge-m3:latest

# 3. 验证模型
ollama list
# 应该能看到：
# NAME                    ID           SIZE    MODIFIED
# deepseek-r1:1.5b       xxx          1.1 GB  xxx
# bge-m3:latest          xxx          1.2 GB  xxx
```

### 2.3 安装 RAGFlow

详细步骤参考 [RAGFlow 安装部署指南](/posts/ragflow/)

**关键点提醒：**
- ⚠️ 使用完整版镜像，不要用 slim 版本
- ⚠️ 确保显卡驱动已更新（GPU 加速）

```bash
# 克隆项目
git clone https://github.com/infiniflow/ragflow.git
cd ragflow

# 切换到稳定版本
git checkout v0.16.0

# 启动服务
cd docker
docker compose up -d

# 等待启动完成（约 5-10 分钟）
docker compose ps
```

## 三、配置模型

### 3.1 登录 RAGFlow

访问 http://localhost:9380，注册管理员账号并登录。

### 3.2 添加聊天模型

1. 点击右上角头像 → **模型管理**
2. 点击 **添加模型**
3. 填写配置：

```
模型类型：Ollama
模型名称：DeepSeek R1 1.5B
Base URL：http://host.docker.internal:11434
模型标识：deepseek-r1:1.5b
```

4. 点击 **保存**

### 3.3 添加 Embedding 模型

同样步骤添加 Embedding 模型：

```
模型类型：Ollama
模型名称：BGE-M3
Base URL：http://host.docker.internal:11434
模型标识：bge-m3:latest
```

### 3.4 验证模型可用

点击模型列表中的 **测试** 按钮，确认能正常返回结果。

## 四、准备知识库文档

### 4.1 文档准备

准备《考勤管理制度》文档，建议格式：

```
knowledge_base/
├── 考勤管理制度.docx      # 主文档
├── 考勤管理制度.txt        # 纯文本备份
└── 常见问题FAQ.txt        # 补充材料
```

**文档内容示例：**

```markdown
# 考勤管理制度

## 第一章 总则
本制度适用于公司全体正式员工...

## 第二章 工作时间
2.1 标准工时
- 上午：09:00 - 12:00
- 下午：13:30 - 18:00

## 第三章 考勤规则
3.1 打卡要求
- 每日需完成上下班两次打卡
- 忘记打卡需在 24 小时内补卡

3.2 迟到早退
- 迟到 15 分钟以内：每次扣款 50 元
- 迟到 15-30 分钟：每次扣款 100 元
- 迟到 30 分钟以上：按旷工半天处理

## 第四章 请假制度
4.1 年假
- 工作满 1 年：5 天
- 工作满 10 年：10 天
- 工作满 20 年：15 天
...
```

### 4.2 文档优化建议

1. **结构清晰**：使用标准的标题层级（H1/H2/H3）
2. **内容完整**：覆盖常见问题场景
3. **格式规范**：避免乱码、特殊字符
4. **大小适中**：单个文档不超过 10MB

## 五、创建知识库

### 5.1 创建知识库

1. 左侧菜单 → **知识库**
2. 点击 **新建知识库**
3. 填写信息：

```
名称：考勤管理制度
描述：公司考勤管理制度文档，包含工作时间、打卡规则、请假制度等
Embedding 模型：BGE-M3
```

4. 点击 **确定**

### 5.2 上传文档

1. 进入刚创建的知识库
2. 点击 **上传文件**
3. 选择准备好的文档
4. 等待解析完成

**解析时间参考：**
| 文档类型 | 大小 | CPU 解析时间 | GPU 解析时间 |
|---------|------|-------------|-------------|
| 纯文本 .txt | 100KB | 10 秒 | 5 秒 |
| Word .docx | 500KB | 30 秒 | 15 秒 |
| 扫描件 PDF | 5MB | 5-10 分钟 | 1-2 分钟 |

### 5.3 检查解析结果

点击文档查看分块结果，确保：
- 内容完整，没有截断
- 分块合理，语义完整
- 表格、列表正确识别

## 六、创建聊天助手

### 6.1 创建助手

1. 左侧菜单 → **聊天**
2. 点击 **新建助手**
3. 配置如下：

```
名称：考勤助手
描述：解答公司考勤管理制度相关问题
关联知识库：考勤管理制度
LLM：DeepSeek R1 1.5B
Temperature：0.1（知识问答建议低温度）
```

### 6.2 配置提示词

在 **系统提示词** 中添加：

```
你是公司的考勤管理助手，专门解答员工关于考勤制度的问题。

回答要求：
1. 基于知识库内容回答，不要编造信息
2. 引用具体条款（如"根据第 3.2 条"）
3. 回答简洁明了，重点突出
4. 如遇不确定的问题，建议咨询 HR 部门

请用中文回答。
```

### 6.3 测试问答

在右侧聊天窗口测试：

```
👤 测试问题 1：上班时间是几点？
🤖 预期回答：根据考勤管理制度，标准工作时间为上午 9:00-12:00，下午 13:30-18:00。

👤 测试问题 2：迟到怎么扣钱？
🤖 预期回答：根据第 3.2 条规定：

🤖 预期回答：根据第 4.1 条，年假天数根据工龄确定：
```

## 七、踩坑记录与解决方案

### 7.1 坑 1：slim 版本没有模型设置

**问题：** 最新版 RAGFlow 默认是 slim 版本，界面上找不到模型配置选项。

**解决：**
```bash
# 切换到完整版
git checkout v0.16.0

# 或修改 .env 文件
RAGFLOW_IMAGE=infiniflow/ragflow:v0.16.0  # 不是 slim
```

### 7.2 坑 2：PDF 解析慢到卡死

**问题：** 图片型 PDF（扫描件）在 CPU 上解析极慢，电脑直接卡住。

**解决：**
1. 更新显卡驱动到最新版本
2. 安装 NVIDIA Container Toolkit
3. 重启 Docker 服务

```bash
# 验证 GPU 可用
docker run --rm --gpus all nvidia/cuda:11.8.0-base-ubuntu22.04 nvidia-smi
```

**效果对比：**
- CPU：10 页扫描件 PDF 解析约 10 分钟，期间电脑几乎无法操作
- GPU：同样文档约 1 分钟完成，不影响正常使用

### 7.3 坑 3：默认 Embedding 模型效果差

**问题：** 使用 RAGFlow 内置的 Embedding 模型，中文检索效果不理想。

**解决：** 切换到本地 Ollama 的 bge-m3 模型

```bash
ollama pull bge-m3:latest
```

在 RAGFlow 中添加模型后，重新解析知识库文档。

**效果对比：**
| 模型 | 中文检索准确率 | 响应速度 |
|------|--------------|---------|
| 内置 BAAI | 70% | 快 |
| bge-m3 (本地) | 90% | 中等 |
| bge-m3 (GPU) | 90% | 快 |

### 7.4 坑 4：Docker 内无法访问 Ollama

**问题：** RAGFlow 容器内访问 `http://localhost:11434` 失败。

**原因：** 容器内的 localhost 指向容器本身，不是宿主机。

**解决：** 使用 `host.docker.internal` 域名

```
Base URL: http://host.docker.internal:11434
```

### 7.5 坑 5：回答质量不好

**问题：** DeepSeek R1 1.5B 模型太小，回答有时不够准确。

**临时解决：**
- 降低 Temperature 到 0.1
- 优化系统提示词
- 确保知识库文档质量

**长期解决：** 升级到更大的模型
```bash
ollama pull deepseek-r1:7b    # 需要 8GB+ 显存
ollama pull deepseek-r1:14b   # 需要 16GB+ 显存
```

## 八、性能优化

### 8.1 模型配置优化

```yaml
# Ollama 调优
OLLAMA_NUM_PARALLEL=4      # 并行请求数
OLLAMA_MAX_LOADED_MODELS=2 # 最大加载模型数
```

### 8.2 RAGFlow 配置优化

编辑 `docker/.env`：

```bash
# 增加 Worker 数量
WORKER_COUNT=4

# 调整 Chunk 大小
CHUNK_SIZE=512
CHUNK_OVERLAP=50
```

### 8.3 硬件优化建议

| 组件 | 当前配置 | 推荐升级 |
|------|---------|---------|
| 内存 | 16GB | 32GB |
| 显卡 | 无 / 集成显卡 | RTX 3060 12GB+ |
| 硬盘 | HDD | NVMe SSD |

## 十、总结

### 项目成果

| 指标 | 使用前 | 使用后 |
|------|-------|-------|
| HR 日均咨询量 | 50+ | 10 |
| 平均响应时间 | 30 分钟 | 即时 |
| 员工满意度 | - | 提升 40% |

### 关键经验

1. **版本选择很重要**：不要用 slim 版本，功能不全
2. **GPU 是必需品**：CPU 跑不动图片型 PDF
3. **本地模型更靠谱**：bge-m3 比默认 Embedding 效果好很多
4. **文档质量决定效果**：输入垃圾，输出也是垃圾

### 技术栈总结

```
前端：RAGFlow Web UI
后端：RAGFlow API + RAG Engine
模型：Ollama + DeepSeek R1 1.5B + bge-m3
存储：Elasticsearch + MinIO
部署：Docker Compose
```
